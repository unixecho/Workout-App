# I18N.md — RepUp localization architecture

Owner decision (2026-07-12): **Hebrew is the home market.** Israeli users are
the primary audience; English-preferring users (in Israel or abroad) are a
bonus, served automatically. This document is the architecture for that —
read it before touching any user-facing string.

## 1. Market model & goals

| | |
|---|---|
| Primary locale | `he` (Hebrew, Israel) — full RTL |
| Secondary locale | `en` (English) — always complete, it is the source catalog |
| Formatting region | Israel for both languages (day-first dates, Sunday-first week, kg, Western digits) |
| Non-goals (now) | Arabic, Russian, French; URL-based locales; per-field user language mixing |

Quality bar: a Hebrew-speaking user should never *feel* the app was written
in English and translated. That means native word order (message functions,
never string concatenation), Israeli gym vocabulary (§5), Sunday-first
calendars (§7), and a full RTL pass per screen (§6).

## 2. Locale resolution

Priority, resolved per request in `src/lib/i18n/server.ts`:

1. **Explicit choice** — `repup-locale` cookie, set only by the Profile →
   Language switcher (server action `setLocale`). One year, `sameSite: lax`.
2. **Negotiated** — `Accept-Language` header, highest-q supported language
   wins. `iw` (legacy Android tag for Hebrew) counts as `he`.
3. **Home-market default** — `he`. Reached only when the browser names
   neither Hebrew nor English (e.g. a Russian-only device in Ra'anana).

No cookie is written during negotiation — an OS language change keeps
working until the user picks a language explicitly. **Why not URL locales
(`/he/...`)**: the app is fully behind auth; there is no SEO or shareable-
link reason to encode language in the URL, and stable URLs mean zero churn
in auth callbacks, redirects, and realtime links.

`<html lang dir>` is set once in the root layout from the resolved locale;
client components receive it via `I18nProvider` (only the locale string
crosses the RSC boundary — dictionaries contain functions).

## 3. String catalog

- `src/lib/i18n/dictionaries/en.ts` is the **source of truth**;
  `Dictionary = typeof en`. `he.ts` must satisfy `Dictionary`, so a missing
  or extra key is a **compile error** — coverage is enforced by `tsc`, not
  by discipline.
- Keys are organized **by screen** (`today.*`, `player.*`, `onboarding.*`),
  with `common.*` for genuinely shared strings only. Don't force sharing:
  two screens saying "Save" may diverge in Hebrew register later.
- **Messages with values are functions**, never template strings:
  `sessionsThisWeek: (done, planned) => …`. Hebrew word order and
  inflection must never be constrained by an English sentence skeleton.
- **Plurals** are handled inside each language's message function. Hebrew
  has dual forms and irregular counting (יום/יומיים/3 ימים; שבוע/שבועיים)
  — write them out per message. If a message accumulates more than 3 plural
  branches, switch that message to `Intl.PluralRules("he-IL")` internally.
  No ICU library: v1 is ~350 strings and two locales; a typed module beats
  a dependency (revisit at locale #3).

## 4. Hebrew register & gender (the "professional" part)

Hebrew verbs are gendered; addressing everyone in masculine imperative
("התחל אימון") reads dated. RepUp's voice:

- **Prefer impersonal/inclusive constructions** — the standard for modern
  Israeli product copy: "מתחילים?" not "התחל". Present-tense plural
  ("מתאמנים", "ממשיכים") and noun phrases ("האימון הבא") avoid gender
  entirely and sound native.
- Where a direct address is unavoidable and no neutral phrasing works,
  use masculine singular (current Israeli convention) — and log the key in
  a `// GENDERED` comment in `he.ts`. If these accumulate, a profile-level
  preferred-address setting is the v2 fix (the catalog structure already
  supports it: swap the message function).
- Exclamation marks translate poorly — Hebrew motivational copy runs
  warmer and shorter ("יפה!", "עוד סט אחד"). Translate intent, not words.

## 6. RTL strategy

`<html dir="rtl">` flips flex/grid/text automatically. The residue is:

1. **Physical CSS in inline styles** — as each screen is extracted, its
   `paddingLeft/marginRight/left/right/textAlign` become logical
   (`paddingInlineStart`, `insetInlineEnd`, `textAlign: "start"`, …).
   React supports all logical properties camelCased. **Rule: an extracted
   screen must contain no physical horizontal properties** unless visually
   direction-fixed (e.g. chart geometry).
2. **Directional glyphs** — chevrons `›`, back arrows: add class
   `dir-flip` (globals.css mirrors it under `[dir="rtl"]`).
3. **LTR islands** — numbers with units ("12 kg"), emails, handles: Hebrew
   text renders them correctly via the Unicode bidi algorithm in almost all
   cases; wrap in `<bdi>`/`dir="ltr"` only when a real rendering bug shows
   up in the audit, not preemptively.
4. **Charts stay LTR** (time flows left→right even in Hebrew data viz —
   standard Israeli convention), but their *labels* are translated.
5. **Per-screen RTL audit** is a release gate: toggle to Hebrew, walk the
   screen, check alignment/chevrons/scroll/animation directions
   (`translateX` in gesture code needs logical inversion — the session
   editor drag code is index-based, unaffected).

## 7. Formats

- **Dates/numbers**: always through `src/lib/i18n/format.ts`
  (`formatDate`, `formatNumber`) — never bare `toLocaleDateString(undefined, …)`,
  which hydrates inconsistently. Tags: `he-IL`, and `en-GB` for English so
  dates are day-first as Israelis expect even in English.
- **Week starts Sunday in Hebrew UIs.** Canonical storage everywhere stays
  Monday-first indexes (0=Mon…6=Sun — `mondayIndex`, `weekday_availability`,
  plan generation are untouched). Display-time reordering only, via
  `weekdayDisplayOrder(locale)`. Applies to: onboarding day picker,
  availability sheet, Plan month grid, week strips.
- **Digits**: Western (standard in Israel). **Units**: kg canonical.

## 8. Database content (exercise library)

UI strings live in the app bundle; **content** (105 exercises: names, form
cues, common mistakes, adaptation notes; badge names/descriptions) lives in
Postgres and is translated in data, not code:

- `exercise_translations (exercise_slug, locale, name, form_cues text[],
  common_mistakes text[], note_overrides jsonb, PRIMARY KEY (exercise_slug, locale))`
  — same shape for `badge_translations`. English stays in the base tables
  (existing seeds untouched); Hebrew rows ship as data migrations in
  reviewable batches (~25 exercises each). Readers `LEFT JOIN` on
  (slug, locale) and fall back per-field with `COALESCE`.
- **Plan-day titles** ("Push — Chest & Triceps") are generated from
  templates in `generate.ts`; the stored English `session_title` remains
  the fallback, and a render-time key map translates known titles. When a
  template changes, its key changes — never parse titles.
- Translation of content is a **content job with a native reviewer**, not
  a code job; the schema above lets it ship on its own cadence.
