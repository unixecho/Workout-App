# I18N.md — Hebrew locale (branch: i18n-hebrew)

Status: **groundwork, local branch only — owner asked not to push Hebrew
translations yet.** This doc travels with the branch.

## Approach (decided)

- **Locale is a cookie (`repup-locale`), not a URL segment.** The app is
  fully behind auth — there is no SEO reason for `/he/...` routes, and
  keeping URLs stable means zero churn in links, redirects, or the auth
  callback. `src/lib/i18n/` owns everything.
- **Hand-rolled typed dictionary, no dependency.** `en.ts` is the source of
  truth (`Dictionary = typeof en`); `he.ts` must satisfy it, so a missing
  key is a compile error. Messages needing values are *functions*
  (`minutes(n)`), so Hebrew word order is never constrained by template
  strings. If plural rules get hairy later, reach for `Intl.PluralRules`
  before reaching for a library.
- **Server components**: `getI18n()` from `src/lib/i18n/server.ts`.
  **Client components**: `useI18n()` from `src/lib/i18n/client.tsx` — only
  the locale string crosses the RSC boundary (dictionaries hold functions,
  which don't serialize).
- **RTL**: `<html dir="rtl">` flips automatically for Hebrew (root layout).
  Flex/grid layouts flip for free. The audit that remains: physical inline
  styles (`paddingLeft`, `marginRight`, `left:`/`right:` offsets,
  directional chevrons `›` and back-arrows) used across components — those
  need logical equivalents or `dir`-aware values screen by screen.
- **Switcher**: Profile → Account → Language (tap cycles EN ↔ HE), server
  action sets the cookie + revalidates the layout.

## Done on this branch

- Infra: `src/lib/i18n/{index.ts,server.ts,client.tsx,actions.ts,dictionaries/}`
- Root layout sets `lang` + `dir` from the cookie
- Translated: Tab bar, Session Editor (full screen), Profile language row
- Hebrew catalog for all of the above

## Remaining (rough order)

1. Extract the rest of the screens into the dictionary (Today, Plan,
   Onboarding, Workout Player, Library, Stats, Badges, Friends, Profile) —
   mechanical but large; do a screen per sitting and keep `en.ts` organized
   by screen.
2. RTL audit pass per screen (see above) — check every screen with the
   toggle on; fix physical paddings/absolute offsets/chevrons.
3. Dates/numbers: `toLocaleDateString(undefined, …)` calls should pass the
   active locale (e.g. `he-IL`) — grep `toLocale`.
4. **DB content** (exercise names, form cues, mistakes, adaptation notes):
   add an `exercise_translations (slug, locale, name, form_cues,
   common_mistakes, adaptations)` table keyed by slug+locale with English
   fallback — keeps the seed migrations locale-free and lets translations
   ship independently. Plan-day titles ("Push — Chest & Triceps") come from
   `generate.ts` templates and also need keys.
5. Hebrew font sanity check (system stack renders Hebrew fine; verify
   weights 600–800 don't fake-bold on iOS).

## Why not next-intl

Cookie-mode next-intl would work, but v1 needs ~200 strings, no ICU
plurals yet, and no URL routing — a 60-line typed module beats a
dependency the owner would have to trust. Revisit if requirements grow
(pluralization-heavy stats copy, more locales).
